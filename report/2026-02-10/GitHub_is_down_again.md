# 2026-02-10: GitHub 再次宕機

**來源：** Hacker News
**故事 ID：** 46946827
**Hacker News 連結：** https://news.ycombinator.com/item?id=46946827
**網址：** https://www.githubstatus.com/incidents/54hndjxft5bx
**分數：** 484 | **評論數：** 385

## 摘要

GitHub 於 2026 年 2 月 9 日發生服務中斷事件，主要影響通知系統。根據 GitHub 官方狀態頁面，此次事件導致通知延遲長達 1 小時 20 分鐘，經過數小時的修復後才完全恢復正常。這不是孤立的個案，而是 GitHub 近期頻發穩定性問題的最新一起事件。

## 為什麼有趣

這是一個**值得關注的產品穩定性事件**，原因如下：

1. **行業基礎設施級別的重要性**：GitHub 已成為全球軟體開發的基礎設施，承載著數百萬開發者的代碼、CI/CD 流程、Issues 追蹤等核心工作流

2. **頻發的穩定性問題**：社群反應顯示這已經不是偶發事件，而是 Azure 架構遷移後的長期問題

3. **企業級信任危機**：許多企業的 SOC2 等合規要求明確指定使用 GitHub，服務穩定性直接影響業務連續性

4. **對微務務架構的反思**：事件引發關於過度複雜化基礎設施是否值得的技術討論

## 主要討論點

### 1. 架構遷移與 Azure 依賴

- 微軟收購後強制 GitHub 遷移至 Azure 架構
- 多數近期事件直接歸因於 Azure 的穩定性問題
- 社群觀察：遷移過程中，數據和服務處於「半遷移」狀態，增加了系統複雜性和故障面

### 2. 穩定性記錄與 SLA 質疑

- 用戶回憶舊版 GitHub Status 頁面曾真實顯示運行時間，後被替換
- GitHub 的「單個 9」（99%）整體可用性已不足以稱為基礎設施級服務
- Copilot 服務穩定性最差（低於 99%），但 Git 操作也頻受影響

### 3. 企業依賴與工作流風險

- 許多團隊的工作完全依賴 GitHub（ticketing、boards、workflows、CI/CD）
- GitHub Actions 失敗可能導致部署流程無觸發，需要人工監控
- 問題：開發者生產力直接受影響，無法正常提交 PR 或進行 Code Review

### 4. 對基礎設施可用性的反思

**觀點 A（高可用性必需）**：
- Amazon、Google、Shopify 等公司為了可靠性願意承擔額外複雜性
- 技術公司的開發者需要高可用性服務保持生產力
- 作為收費託管服務，GitHub 應提供更高可靠性

**觀點 B（過度工程化）**：
- Git 本身是分散式技術，理論上不需要 24/7 雲端服務可用性
- 社群過度依賴中心化服務（Issues、PR、CI），這是流程設計問題，而非服務問題
- 早期互聯網時代沒有這種高可用性要求，開發工作照樣進行

### 5. 替代方案與分散式協作

- **GitLab**：有用戶正在研究遷移方案
- **分散式問題追蹤**：如 `git-bug` 將 issues 和 PRs 存儲在 git 倉庫本身，不依賴中心化服務
- **Go module cache**：Google 的模組緩存可作為 GitHub 下載故障的備選方案

### 影響範圍

雖然本次主要是通知系統，但社群反應顯示：
- GitHub Actions 觸發可能受影響
- Webhook 交付延遲影響 CI/CD 流程
- 部分用戶報告 Web UI 響應緩慢

## 社群共識與爭議

### 共識
- GitHub 近期的穩定性確實在下降
- Azure 遷移是主要原因之一
- 企業級用戶對此日益不滿

### 爭議
- **是否應追求「多個 9」的可用性**：一派認為作為基礎設施必須高可用；另一派認為過度工程化，應接受偶發停機
- **責任歸屬**：是 GitHub 自身架構問題，還是微軟 Azure 的問題？

## 評價

**值得關注 / 有趣**

這個故事反映了雲端基礎設施時代的一個重要議題：當全球軟體開發高度依賴單一中心化平台時，該平台的穩定性問題會產生連鎖反應。GitHub 從「開發者的工具」演變為「軟體開發基礎設施」，但可靠性和運營標準似乎沒有同步提升。

關鍵啟示：
1. **供應鏈風險**：過度依賴單一平台是企業風險
2. **分散式的價值**：Git 的分散式設計哲學提醒我們，不應將所有信任放在中心化服務上
3. **遷移代價**：微軟強制 GitHub 遷移至 Azure 的代價正在顯現
4. **SLA 期望**：付費企業用戶有權要求更高的服務可用性承諾
