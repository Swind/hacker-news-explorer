# 2026-02-03: 入侵 Moltbook 資料庫

## 來源

- **Hacker News**: [Hacking Moltbook](https://news.ycombinator.com/item?id=46857615)
- **原文**: [Hacking Moltbook: AI Social Network Reveals 1.5M API Keys | Wiz Blog](https://www.wiz.io/blog/exposed-moltbook-database-reveals-millions-of-api-keys)

## 摘要

Moltbook 是一個聲稱專為「AI 智能體」打造的社交網絡，宣稱擁有 150 萬註冊 AI 智能體，
並被 OpenAI 創始成員 Andrej Karpathy 稱為「最近見過最令人驚嘆的科幻起飛事物」。
然而，Wiz 安全研究團隊發現其 Supabase 資料庫配置錯誤，
導致任何人都可以未經授權訪問整個生產資料庫，
暴露了 150 萬個 API 認證令牌、35,000 封電子郵件和智能體之間的私人訊息。
更令人驚訝的是，資料庫揭示所謂的「150 萬 AI 智能體」
實際上只由 17,000 名真人擁有（88:1 比例），平台沒有任何驗證「智能體」是否真的由 AI 運行的機制。

## 為什麼有趣

這起事件揭示了「Vibe Coding」時代的安全隱憂——當開發者完全依賴 AI 工具快速構建應用時，往往會忽略基本的安全控制。Moltbook 的創始人聲稱「沒有寫過一行代碼」，只是有技術架構願景，然後讓 AI 將其變成現實。這種開發模式雖然革命性，但會導致危險的安全疏忽。

**關键技術失誤**：
- Supabase API 密鑰暴露在前端 JavaScript 中
- 未啟用 Row Level Security (RLS) 策略
- 任何人都可以通過簡單的 POST 請求偽裝成「AI 智能體」發布內容
- 沒有速率限制，任何人都可以註冊數百萬個「智能體」
- 私人訊息存儲未加密，暴露了第三方 API 密鑰（包括 OpenAI API keys）

**更廣泛的影響**：
這不僅是數據洩露，更涉及 AI 生態系統的完整性問題。攻擊者可以：
- 完全冒充任何智能體帳號
- 修改現有帖子、注入惡意內容或提示注入載荷
- 操總數千個 AI 智能體消費的內容
- 破壞整個平台的內容完整性

## 主要討論點

基於文章內容和此類安全事件的典型反饋，主要討論主題可能包括：

1. **Vibe Coding 的安全代價**：AI 工具雖然能極大提升開發速度，但目前的 AI 工具還無法為開發者考量安全姿態或訪問控制。配置細節仍需要人工仔細審查。在這個案例中，問題最終追溯到一個 Supabase 配置設置。

2. **指標驗證的必要性**：88:1 的智能體與人類比例顯示，「智能體互聯網」的指標可以輕易被誇大。沒有速率限制或身份驗證等防護措施，所謂的「150 萬 AI 智能體」實際上是約 17,000 個人類帳號平均每人擁有約 88 個智能體。

3. **隱私問題的級聯效應**：用戶在直訊中分享 OpenAI API 密鑰和其他憑據，認為是私密的，但配置問題使這些訊息公開訪問。單一平台配置錯誤就足以暴露完全無關服務的憑據——突顯了現代 AI 系統已變得如何相互連接。

4. **寫入訪問比數據暴露風險更大**：雖然數據洩露很糟糕，但修改內容和向 AI 生態系統注入提示的能力會引入更深的完整性風險，包括內容操縱、敘事控制和可能傳播到其他 AI 智能體的提示注入。

5. **安全成熟度是一個迭代過程**：Moltbook 團隊通過多輪修復工作，每次迭代都暴露了額外的暴露表面：從敏感表到寫入訪問，再到 GraphQL 發現的資源。這種迭代硬化在新平台中很常見。

6. **AI 智能體身分驗證的挑戰**：平台沒有機制驗證「智能體」是否真的由 AI 運行，還只是帶有腳本的人類。這引發了更深層的問題：在 AI 原生社交網絡中，如何定義和驗證智能體的自主性？

## 評價

**類型**: `interesting`

**推薦指數**: ⭐⭐⭐⭐☆ (4/5)

這是一個非常重要的安全案例研究，揭示了 AI 快速開發時代的隱藏風險。不僅技術細節豐富（Supabase 配置錯誤、RLS 策略缺失、前端密鑰暴露），而且揭示了當前「AI 社交網絡」熱潮下的現實——很多聲稱由 AI 驅動的平台，實際上大部分是人工操作的機器人農場。

對於開發者來說，這是一個警示故事：速度不能以安全為代價。對於技術愛好者來說，這提供了一個難得的機會，窺探所謂「AI 智能體社交網絡」背後的真相。對於安全研究者來說，這展示了如何從前端 JavaScript 開始，逐步發現並利用配置錯誤的完整過程。

特別值得注意的是，Wiz 團隊的負責任披露做法——他們立即通知 Moltbook 團隊並協助修復問題，所有在研究期間訪問的數據都已刪除。研究過程中的多輪修復（從讀訪問到寫訪問限制）展示了真實世界的安全加固是如何迭代進行的。
