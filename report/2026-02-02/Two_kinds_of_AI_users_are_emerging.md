# 2026-02-02: Two kinds of AI users are emerging

**來源：** Hacker News
**故事 ID：** 46850588
**Hacker News 連結：** https://news.ycombinator.com/item?id=46850588
**網址：** https://martinalderson.com/posts/two-kinds-of-ai-users-are-emerging/
**分數：** 482 | **評論數：** 233

## 摘要

本文探討了 AI 使用者群體正在出現的**顯著分化現象**。作者觀察到兩類截然不同的使用者：第一類是「**power users**」──包括許多非技術人員在內，他們積極採用 Claude Code、MCP、skills 等進階 AI 工具；第二類則主要透過 ChatGPT 等**聊天介面**與 AI 互動。作者強調，**Microsoft 365 Copilot** 在企業環境中的普及實際上造成了 AI 採用的**能力陷阱**，而這種分化正在對企業的長期競爭力產生深遠影響。

## 為什麼有趣

### 1. 重要的產業趨勢觀察
這篇文章揭示了一個**正在加速進行的數位鴻溝**：
- **小型企業**因為技術債務較少、限制較少，能夠快速採用進階 AI 工具
- **大型企業**受制於安全性考量、遺留系統和 IT 政策，只能使用功能受限的 AI 工具（如 Copilot）

這可能導致「**小團隊擊敗千倍規模的大公司**」的歷史性轉變。

### 2. Microsoft Copilot 的嚴重問題
作者提出了一個**令人震驚的觀察**：
- Microsoft 自己內部正在推廣使用 Claude Code，而不是自己的 Copilot
- 這儘管 Microsoft 擁有 OpenAI 的大量股權，且 Copilot 幾乎免費提供給員工
- Copilot 的「agent」功能被形容為「**徹頭徹尾的笑話**」，與 CLI coding agents 相比差距巨大

### 3. 非技術人員的技術轉型
一個**反直覺的發現**：
- Finance 等非技術角色的使用者正在從 Claude Code 中獲得**巨大價值**
- 這些人學會使用 Python 來擺脫 Excel 的限制
- 一位非技術主管在幾次提示內就將 30 個 sheet 的複雜 Excel 財務模型轉換為 Python，甚至還自動加入了 unit tests

### 4. 企業面臨的生存風險
作者警告這對許多企業來說已成為「**存在性風險**」：
- 高層決策者使用效果不佳的 AI 工具，進而對 AI 產生錯誤判斷
- 企業花費巨資諮詢卻無法取得實質進展
- 無法使用 sandboxed agents 和 API 存取，導致競爭力大幅落後

## 主要討論點

### 討論主題一：AI 使用者分類的多維度

評論區提出了多種不同的分類方式：

**1. 工具使用者 vs. 思維外包者**
- **工具使用者**：將 AI 視為實習生/瑣碎任務執行者，了解其限制
- **思維外包者**：將整個技能集和思考外包給 AI，對主題了解不多，只在乎結果

**2. 兩代 AI 工具的區分**
- **第一代**：簡單的聊天介面（ChatGPT 網頁版）
- **第二代**：具備 RAG 和 agent 能力的工具（Claude Code、MCP）

**3. 批評假二分法**
- 許多評論認為原文的分類過於簡化
- 真實情況是連續光譜，而非二元對立
- 有些使用者會根據不同情境在兩種模式間切換

### 討論主題二：企業 AI 採用的結構性障礙

評論區補充了許多關於企業困境的見解：

**IT 政策的限制**
- 環境高度鎖定，無法運行基本腳本直譯器
- 遺留系統缺乏 API，agents 無法連接
- 工程部門高度 siloed，甚至完全外包

**安全與生產力的矛盾**
- 確實不希望員工用 coding agents 任意操作生產資料庫
- Sandbox agents 的技術難度很高
- 缺乏能夠建置安全基礎設施的內部團隊

**「API-first」的重要性**
- 擁有內部 API 的企業將能做更多事
- 這可以是 readonly data warehouse，或是完整的業務流程 API
- 傳統企業 SaaS 通常不是「API-first」產品

### 討論主題三：工作文化的變遷

一條**深刻的評論**反映了一線工程師的困境：

> 「我開始在工作中外包思考，因為我的公司明確表示他們不想要/付不起思考工程師。思考需要時間，但他們想要快速交付。」

這引發了關於：
- 職業倦怠和「安裝應用程式」文化
- AI 可能加劇而非解決企業的短期主義問題
- 員工因為重複學習新技術而感到精疲力竭

### 討論主題四：非技術使用者的成功案例

一位**德語學習者**分享了令人印象深刻的經驗：
- 使用 Claude Code 和 ElevenLabs API 建立德語聽力練習應用
- 生成對話和問題
- 德語老師對結果印象深刻
- 關鍵在於：**明確知道 AI 的限制**，將其視為達成目標的手段而非目的

這證明了：**非技術人員在掌握正確工具後，能夠實現驚人的生產力躍升**。

### 討論主題五：未來工作形態的預測

作者提出四個未來趨勢：

1. **自下而上的創新**：真正的生產力提升來自小團隊自主嘗試，而非自上而下的 AI 策略
2. **API 成為關鍵**：有內部 API 的公司將遠超沒有的公司
3. **安全機制**：需要妥善設計的 VM 和網路限制來安全地運行 agents
4. **遺留系統的鎖定風險**：舊的 SaaS 供應商可能成為創新的瓶頸

評論區補充：
- 「The user prompts, the agent synthesises」的模式可能取代大部分生產力應用
- 擁有 bash sandbox + 程式語言 + API + agent harness，就能構建幾乎任何報表或工具

## 評價

**⭐⭐⭐⭐⭐ 值得深度閱讀**

這篇文章揭示了 AI 採用浪潮中一個**極為重要但被低估的現象**。作者基於真實觀察，提出了許多**反直覺但合理的見解**：

**優點：**
- 提供了具體的案例和對比（Excel → Python 的轉換）
- Microsoft 內部使用 Claude 的觀察尤其具說服力

3. **非技術人員的技術賦能**：擁有正確工具的非技術人員可能成為最強大的 AI 使用者

### 討論持續升溫的主要原因

本文的討論熱度顯著提升（評論數增長約 32%），反映了幾個重要趨勢：

1. **Microsoft 產品策略的爭議持續發酵**
   - 許多評論者分享了使用 Microsoft 365 Copilot 的挫折經驗
   - 有微軟員工匿名證實內部確實在測試多種 AI 工具，包括 Claude
   - 這凸顯了大型軟體公司在「對外銷售」與「內部使用」之間的產品落差

2. **「思維外包」的倫理與職業辯論深化**
   - 討論延伸至「依賴 AI 是否會導致技能退化」的議題
   - 有開發者擔憂：過度依賴 AI 會讓新一代工程師失去理解底層原理的能力
   - 另一方則認為：AI 降低了技能門檻，讓專家能專注於更高階的問題

3. **API 文化和基礎設施的重要性獲得更多共識**
   - 許多 IT 專業人員分享了他們組織在 API 化過程中的挑戰
   - 有系統架構師指出：「API-first 不再是可選，而是 AI 時代的生存必要條件」
   - 討論了 GraphQL vs REST 在 AI agent 整合中的優劣

### 新增的重要見解

**見解一：AI 工具採用與公司規模的關係**

評論區補充了更多數據點：
- **10 人以下公司**：幾乎沒有限制，直接使用最強大的工具
- **50-200 人公司**：開始出現 IT 政策，但仍允許例外申請
- **500 人以上公司**：通常只有「核准清單」上的工具，而 Claude/Code agents 很少在清單上

**見解二：財務與金融業的特殊案例**

多位金融業從業者分享了類似經驗：
- Excel 確實是最大的生產力瓶頸，特別是當模型跨 30+ 個工作表時
- Python + pandas 的靈活性讓金融分析師能做「Excel 永遠做不到的事」
- 一位 quant 分享：他們團隊已經全面禁止提交 Excel 模型，必須用 Jupyter notebooks

**見解三：安全沙箱的實作方案**

技術社群討論了幾種可行的企業級安全方案：
1. **Firecracker microVMs**：AWS Lambda 的底層技術，可隔離每個 agent 執行環境
2. **gVisor**：Google 開發的使用者空間 kernel，提供 process-level 隔離
3. **Kubernetes + Network Policies**：限制 outbound 連線，只允許訪問特定內部 API

一位 DevOps 工程師總結：「安全性不是二元的，而是梯度。我們可以在受控環境下允許 agents 存取特定的 readonly data warehouses。」

**見解四：工作性質的變遷**

一條引起共鳴的評論：

> 「在 AI 之前，年輕工程師的價值在於他們願意做那些重複、瑣碎的任務。現在這些任務被 AI 接管了，他們需要更快找到新的價值定位。」

這引發了對於「入門級工作消失」的擔憂：
- 如果 AI 能做到初級工程师 80% 的工作，那初級工程师如何學習成為資深工程师？
- 有人提出「師徒制」的回歸，但企業認為這「成本太高」

**見解五：Microsoft 的回應（非官方）**

有自稱 Microsoft 產品團隊成員的評論（未驗證）解釋：
- Copilot 的設計目標與 Claude Code 不同
- Copilot 優先考慮「零設定」和「企業合規」
- Claude Code 則是為「願意花時間設定」的 power users 設計
- 兩者在設計哲學上有本質差異，而非單純的品質差距

評論區的反應：承認設計目標不同，但批評 Microsoft 沒有提供「power user mode」給有需要的企業用戶。

### 更新的趨勢判斷

基於新增的 74 則評論，本文揭示的趨勢變得更加清晰：

1. **「AI 鴻溝」正在加速擴大**，而非縮小
   - 小公司的優勢比作者描述的更明顯
   - 大企業的政策僵化讓他們在 AI 競賽中越來越落後

2. **Microsoft 處於尷尬位置**
   - 擁有 OpenAI 股權，但內部卻使用競爭對手的工具
   - 這反映了「企業級」與「個人效能級」產品的根本矛盾

3. **2025-2026 可能是分水嶺**
   - 採用進階 AI 工具的小團隊將開始展現壓倒性生產力優勢
   - 拒絕或無法採用這些工具的組織將在人才市場和商業競爭中受損
