# 2026-02-06: The RCE that AMD won't fix

**來源：** Hacker News
**故事 ID：** 46906947
**Hacker News 連結：** https://news.ycombinator.com/item?id=46906947
**網址：** https://mrbruh.com/amd/
**分數：** 167 | **評論數：** 68

## 摘要

研究人員發現 AMD AutoUpdate 軟體存在一個嚴重的遠端程式碼執行 (RCE) 漏洞：該軟體透過未加密的 HTTP 下載驅動程式更新，且未驗證下載檔案的數位簽章，導致攻擊者可透過 MITM 攻擊替換成惡意執行檔。AMD 將此漏洞回報標記為「超出範圍」(out of scope) 並拒絕修復。

## 為什麼有趣

這是一個**具有重要安全意義的披露**，原因如下：

1. **嚴重的安全漏洞**：AMD AutoUpdate 預裝在數百萬台電腦上，透過 HTTP 下載執行檔且無簽章驗證，形成完美的攻擊向量
2. **供應鏈安全問題**：暴露硬體廠商在更新機制上的安全疏忽，類似 SolarWinds 事件的風險
3. **負責披露困境**：研究人員遵循負責披露流程卻被拒絕，引發對漏洞賞金計畫運作的質疑
4. **企業可能採取行動**：評論指出大型企業可能會考慮全面禁止 AMD 硬體或其更新軟體

## 主要討論點

### 漏洞嚴重性

- **MITM 攻擊易於執行**：攻擊者可在機場咖啡廳設置惡意 Wi-Fi 熱點，攔截 HTTP 流量並注入惡意執行檔
- **影響範圍極廣**：AMD 顯卡驅動程式預裝於大量電腦，潛在受害者數量龐大
- **無沙箱保護**：此類驅動管理軟體難以像其他應用程式一樣被沙箱隔離

### AMD 回應的爭議

- **「超出範圍」標記**：AMD 將 MITM 攻擊列為漏洞賞金計畫的排除項目，但未說明是否會修復問題
- **溝通不當**：回應簡短且不專業，僅寫著「Thank you for hacking our product」，缺乏詳細說明
- **政策 vs. 實際修復**：即使不支付賞金，廠商仍應認真處理可被利用的安全漏洞

### 與 Linux 的對比

- **發行版維護者更可靠**：社群指出免費維護 Linux 發行版的人員似乎比十億美元硬體廠商更重視安全
- **套件管理器模式**：Linux 系統透過套件管理器統一管理驅動程式，避免了低品質的專有軟體

### 技術細節討論

- **Production URL 使用 Development 設定**：程式在生產環境中使用了開發 URL，可能導致其他未知問題
- **多種攻擊向量**：除了 MITM，還可透過 DNS 快取投毒、路由器入侵等方式利用此漏洞

## 評價

**值得關注** - 這是一個重要的安全警訊：

- **高風險漏洞**：影響數百萬台電腦，攻擊門檻極低
- **負責披露的挫敗**：研究人員正確遵循流程卻遭拒絕，反映漏洞賞金計畫的缺陷
- **企業級影響**：大型組織可能重新評估 AMD 硬體的安全性
- **供應鏈安全啟示**：提醒我們關注硬體廠商軟體更新機制的安全性

社群普遍認為此漏洞應被修復，無論是否屬於賞金計畫範圍。此事件可能促使 AMD 重新考慮其立場。
