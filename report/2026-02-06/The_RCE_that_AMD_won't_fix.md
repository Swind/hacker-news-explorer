# 2026-02-06: The RCE that AMD won't fix

**來源：** Hacker News
**故事 ID：** 46906947
**Hacker News 連結：** https://news.ycombinator.com/item?id=46906947
**網址：** https://mrbruh.com/amd/
**分數：** 167 | **評論數：** 68

## 摘要

研究人員發現 AMD AutoUpdate 軟體存在一個嚴重的遠端程式碼執行 (RCE) 漏洞：該軟體透過未加密的 HTTP 下載驅動程式更新，且未驗證下載檔案的數位簽章，導致攻擊者可透過 MITM 攻擊替換成惡意執行檔。AMD 將此漏洞回報標記為「超出範圍」(out of scope) 並拒絕修復。

## 為什麼有趣

這是一個**具有重要安全意義的披露**，原因如下：

1. **嚴重的安全漏洞**：AMD AutoUpdate 預裝在數百萬台電腦上，透過 HTTP 下載執行檔且無簽章驗證，形成完美的攻擊向量
2. **供應鏈安全問題**：暴露硬體廠商在更新機制上的安全疏忽，類似 SolarWinds 事件的風險
3. **負責披露困境**：研究人員遵循負責披露流程卻被拒絕，引發對漏洞賞金計畫運作的質疑
4. **企業可能採取行動**：評論指出大型企業可能會考慮全面禁止 AMD 硬體或其更新軟體

## 主要討論點

### 漏洞嚴重性

- **MITM 攻擊易於執行**：攻擊者可在機場咖啡廳設置惡意 Wi-Fi 熱點，攔截 HTTP 流量並注入惡意執行檔
- **影響範圍極廣**：AMD 顯卡驅動程式預裝於大量電腦，潛在受害者數量龐大
- **無沙箱保護**：此類驅動管理軟體難以像其他應用程式一樣被沙箱隔離

### AMD 回應的爭議

- **「超出範圍」標記**：AMD 將 MITM 攻擊列為漏洞賞金計畫的排除項目，但未說明是否會修復問題
- **溝通不當**：回應簡短且不專業，僅寫著「Thank you for hacking our product」，缺乏詳細說明
- **政策 vs. 實際修復**：即使不支付賞金，廠商仍應認真處理可被利用的安全漏洞

### 與 Linux 的對比

- **發行版維護者更可靠**：社群指出免費維護 Linux 發行版的人員似乎比十億美元硬體廠商更重視安全
- **套件管理器模式**：Linux 系統透過套件管理器統一管理驅動程式，避免了低品質的專有軟體

### 技術細節討論

- **Production URL 使用 Development 設定**：程式在生產環境中使用了開發 URL，可能導致其他未知問題
- **多種攻擊向量**：除了 MITM，還可透過 DNS 快取投毒、路由器入侵等方式利用此漏洞

## 評價

**值得關注** - 這是一個重要的安全警訊：

- **高風險漏洞**：影響數百萬台電腦，攻擊門檻極低
- **負責披露的挫敗**：研究人員正確遵循流程卻遭拒絕，反映漏洞賞金計畫的缺陷
- **企業級影響**：大型組織可能重新評估 AMD 硬體的安全性
- **供應鏈安全啟示**：提醒我們關注硬體廠商軟體更新機制的安全性

社群普遍認為此漏洞應被修復，無論是否屬於賞金計畫範圍。此事件可能促使 AMD 重新考慮其立場。

**1. 文章已被下架**

**2. 企業成本效益分析的深入討論**

部分討論質疑 AMD 的決策邏輯：

> 「AMD 年營收約 300 億美元。將代碼從 HTTP 改為 HTTPS 需要多少工程師時間？1,000 美元？5,000 美元？即使我們極端假設是 10,000 美元，這也只佔 AMD 年營收的 0.00003%，連會計誤差都不到。」

反方觀點指出企業的計算方式不同：
- 不是「修復成本 vs 營收」的比較
- 而是「工程師修復此問題的機會成本 vs 開發新功能的價值」
- 除非修復此問題能帶來更多營收，否則不會被優先考慮

**3. 硬體廠商軟體能力的批評**

多位嵌入式開發者分享了類似經驗：
> 「多年的嵌入式開發經驗讓我相信，大多數硬體公司就是不擅長軟體。我認為部分原因是硬體的長週期開發時間推動了瀑布式開發文化。」

**4. Linux 系統的優勢再受強調**

- 發行版維護者（免費工作）比十億美元硬體廠商更重視安全
- Linux 將驅動程式整合到內核中，消除了對低質量驅動管理軟體的需求
- 這類專有軟體難以被沙盒化隔離

**5. 行業普遍問題確認**

確認多家廠商有相同問題：
- **Gigabyte**：自動更新程式依賴 HTTP
- **ASUS**：存在相同問題
- **Apple HomePod**：甚至無法在不允許 HTTP 的情況下設置

### 技術爭議深化：HTTP + 簽名 vs HTTPS + 簽名

討論進一步深入探討了即使使用簽名驗證，HTTP 仍然不安全的原因：

**HTTP 的額外風險：**
- HTTP 堆疊本身可能有漏洞
- 簽名驗證代碼可能有缺陷
- 解析器可能被特製的 HTTP 響應攻擊
- 隱私洩露（攻擊者可知道用戶正在下載 AMD 更新）
- TLS 堆疊比單純的 HTTP 更難被利用

**關鍵論點：**
> 「對於簽名載荷，沒有區別——你信任 <客戶端> 的身份驗證代碼去讀取一個數據塊、一個簽名，並根據公鑰驗證它。這不比許多 TLS 和 HTTP 庫更可信。」

反駁觀點指出：
> 「未加密的 HTTP 總是簡單的...但 TLS 堆疊是比 HTTP 更困難攻擊的目標。」

### 暫時性緩解措施更新

由於 AMD 拒絕修復，社群建議：

1. **完全停用 AMD AutoUpdate 服務**
2. **從 AMD 官網手動下載驅動**（但仍需驗證是否使用 HTTPS）
3. **使用 Linux 系統**（驅動程式由發行版維護）
4. **在受信任的網絡環境中更新**
5. **企業環境考慮封鎖 AMD 的更新伺服器**

### 長期影響預測

1. **企業採購決策**：大型企業可能因安全問題重新評估 AMD 硬體採購
2. **供應鏈安全審查**：更多組織將要求硬體供應商提供更新機制的安全審計
3. **開源替代方案受推崇**：此事件進一步凸顯了開源軟體在安全透明度方面的優勢
4. **法規可能的介入**：類似 SolarWinds 事件後，可能會有法規要求硬體廠商的安全合規

### 評價更新

此事件已從單純的安全漏洞演變為：

1. **企業社會責任問題**：大型科技公司對基本安全的忽視
