# 網頁內容提取指南

## 使用時機
- 需要閱讀文章全文時
- 提取關鍵資訊時
- 處理特殊網站格式時
- 網頁內容過長需要聚焦時

## 使用 read_webpage 工具

```python
read_webpage(url="https://example.com/article")
```

**注意：** 工具會自動提取主要內容，去除廣告和導航元素。

## 不同類型網站的處理

### 1. 技術博客/文章

**特徵：**
- 標題清晰
- 有發布日期
- 代碼範例
- 結構化內容

**提取重點：**
- 標題和副標題
- 核心論點
- 代碼範例
- 結論

**常見格式：**
- Medium、Dev.to、個人博客
- 公司技術博客

### 2. 新聞網站

**特徵：**
- 倒金字塔結構
- 標題、導言、正文
- 可能付費牆

**提取重點：**
- 標題（<h1>）
- 導語段落
- 關鍵事實
- 引述

### 3. PDF 文檔

**使用 read_webpage 時：**
- 工具會自動處理 PDF
- 可能不完整，需要關注：
  - 標題頁資訊
  - 目錄（了解結構）
  - 摘要/結論

### 4. GitHub README

**特徵：**
- Markdown 格式
- 包含代碼示例
- 有徽章（badges）

**提取重點：**
- 專案描述
- 安裝說明
- 使用範例
- 主要功能列表

### 5. 視頻/音頻內容

**無法直接提取，策略：**
- 讀取描述/說明文字
- 查看評論摘要
- 檢查是否有文字稿（transcript）

## 內容提取策略

### 當內容過長時

**策略 1：分段讀取**
```
先讀前 30% → 了解主題
再讀中間 40% → 獲取核心論證
最後讀後 30% → 查看結論
```

**策略 2：關注結構**
- 標題層級（h1, h2, h3）
- 列表項目
- 加粗/斜體文字
- 引用區塊

**策略 3：關鍵詞搜尋**
```
在內容中搜尋：
- "conclusion", "summary", "takeaway"
- "however", "therefore", "importantly"
- 標題中的關鍵詞
```

### 當內容難以理解時

**檢查：**
1. 是否為技術論文？→ 關注摘要和結論
2. 是否為系列文章？→ 尋找第一篇
3. 是否需要背景知識？→ 查看評論解釋

### 當網頁載入失敗時

**可能原因：**
- 需要登入
- 有防爬蟲機制
- 伺服器問題
- URL 錯誤

**應對策略：**
- 檢查 HN 評論中是否有摘要
- 尋找其他來源（鏡像、轉載）
- 跳過該來源，依賴評論討論

## 關鍵資訊識別

### 技術文章關鍵點

**必須提取：**
- 核心技術/概念是什麼？
- 解決了什麼問題？
- 如何實現的（高層次）？
- 有什麼限制或缺點？

**可選提取：**
- 性能數據
- 基準測試結果
- 與替代方案的比較

### 新聞文章關鍵點

**5W1H：**
- Who：誰/什麼組織
- What：發生什麼事
- When：何時發生
- Where：何地發生
- Why：為什麼發生
- How：如何發生

### 評論內容處理

當使用 `read_webpage` 讀取 HN 評論頁時：

**關注高贊評論**
- 通常代表社群共識
- 可能有技術細節
- 可能有不同意見

**識別討論主題**
- 主要論點有哪些？
- 有爭議嗎？爭論焦點是什麼？
- 有專業人士參與討論嗎？

**提取有用資訊**
- 背景補充
- 相關鏈接
- 經驗分享
- 批評意見

## 錯誤處理

### 常見錯誤

```
Error: Unable to fetch webpage
```

**應對：**
1. 確認 URL 正確
2. 檢查是否需要特殊處理（如 PDF）
3. 依賴 HN 評論摘要
4. 記錄在報告中說明無法訪問

### 內容質量評估

**低質量指標：**
- 內容重複
- 廣告過多
- 標題黨
- 無實質內容

**應對：**
- 在報告中註明
- 主要依賴評論討論
- 判斷是否值得分析

## 最佳實踐

✅ **先看結構**：標題、章節、摘要
✅ **關注重點**：提取核心資訊，非逐字閱讀
✅ **結合評論**：HN 評論常有補充資訊
✅ **記錄來源**：註明資訊來源

❌ **不要逐字閱讀**：長文章效率低
❌ **不要忽視結論**：結論通常最重要
❌ **不要迷失細節**：聚焦主要論點
